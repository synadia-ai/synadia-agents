OpenCode multi-session/NATS UX mitigation spike

Research report for Phase 6.5B: whether OpenCode can avoid the current standalone adapter UX regression without throwing away the working Synadia Agent Protocol bridge.

Executive recommendation

Ship the current standalone OpenCode adapter as the immediate PR, but do not pretend the UX concern is solved. Open a follow-up PR/phase for a wrapper launcher plus a multi-session manager.

The adapter already has a working AgentService bridge with managed and attached modes. Blocking it for an in-process plugin rewrite would trade a known protocol adapter for a larger speculative integration. The evidence says OpenCode has enough hooks to make the UX much better, but the least risky mitigation is a launcher/manager that starts or attaches OpenCode sessions and registers NATS identities automatically.

  1. Merge/PR the current adapter with explicit UX limitations.
  2. Follow immediately with opencode-nats launcher UX.
  3. Add adapter-side multi-session registration.
  4. Spike in-process OpenCode plugin registration after wrapper/manager UX is proven.

Questions answered

1. In-process plugin/channel?

Partly yes

OpenCode plugins load automatically from project/global directories or npm config and receive client, project, directory, worktree, serverUrl, and Bun shell. That is enough to spike a NATS registration plugin. MCP alone is not enough because the product need is Synadia protocol discovery/prompting, not just OpenCode tools.

2. Hooks and extension points?

Yes

Server hooks include event, permission.ask, chat.message, command.execute.before, tool.execute.before/after, shell.env, and compaction hooks. TUI plugins expose routes, dialogs, toasts, keymaps, event bus, and client access.

3. Multi-session adapter?

Yes, with manager refactor

The current adapter is single config, one OpenCode client, one NATS connection, one AgentService, one active session id. Multi-session requires a registry that creates/stops one service per selected OpenCode session and hardens permission routing.

4. Wrapper launcher?

Best next step

A wrapper can start opencode serve, wait for health, start opencode-agent in attached mode, and optionally launch opencode attach. It improves UX without rewriting the adapter.

Implementation sketches

Wrapper launcher

opencode-nats start [--tui] [--name NAME] [--session SESSION]
  -> choose/fix OpenCode server host+port
  -> start `opencode serve` or use `--attach`
  -> start `opencode-agent start --base-url http://... --name NAME ...`
  -> optionally launch `opencode attach http://...`
  -> on exit, stop adapter and managed server

Acceptance: child cleanup on SIGINT, fixed-port health wait timeout, auth/password propagation, attached mode does not start a second OpenCode server, README quickstart smoke.

Multi-session manager

class OpenCodeAgentManager {
  async start(): Promise<void>;
  async reconcile(): Promise<void>;
  async stop(): Promise<void>;
}

The manager owns one NATS connection and one OpenCode SDK client, lists/watches OpenCode sessions, and creates one collision-safe AgentService per selected session. Tests must prove cross-session event and permission isolation.

Server plugin spike

export const SynadiaNatsPlugin: Plugin = async ({ client, directory, project, serverUrl }) => {
  const channel = await startNatsRegistration({ client, directory, project, serverUrl });
  return {
    event: async ({ event }) => channel.handleOpenCodeEvent(event),
    "permission.ask": async (input, output) => channel.observePermission(input, output),
    dispose: async () => channel.stop(),
  };
};

Before production, prove plugin reload/dispose behavior, duplicate-registration handling, and NATS reconnect behavior.

TUI bleedDriving /tui/* endpoints can mutate the wrong local prompt/session.Keep protocol prompts on session API; use TUI endpoints only for explicit UX/toasts.
Server/session mismatchopencode serve starts a new server; TUI also starts a server unless attached/fixed-port.Wrapper owns port/base URL and launches opencode attach.

Duplicate subject suffixesMultiple sessions can share title/workspace labels.Use sanitized id/hash suffixes and collision tests.
